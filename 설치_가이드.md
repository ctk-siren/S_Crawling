## 3. Oracle Cloud 서버 만들기

### 3-1. 회원가입
[cloud.oracle.com](https://cloud.oracle.com) 가입. 신용카드 본인확인이 필요하나 **Always Free 자원은 과금되지 않는다.**

⚠️ **홈 영역(Home Region)** 선택 — 가입 중 한 번 정하면 못 바꾼다. 목록에 한국이 없으면 **일본(Tokyo/Osaka)** 을 고른다(한국과 가장 가깝고 무료 한도 동일). 저트래픽이라 리전이 멀어도 체감 차이는 거의 없다.

### 3-2. 인스턴스(VM) 생성
콘솔 → Compute → Instances → **Create Instance**
- **Image**: Canonical **Ubuntu** (Minimal 말고 일반 버전 권장)
- **Shape**: **Always Free Eligible** 표시가 있는 것 (VM.Standard.E2.1.Micro 또는 Ampere A1)
- **SSH keys**: **Save private key** 로 개인키(.key) 다운로드 — ⚠️ 잃어버리면 접속 불가
- ⚠️ **Networking**: **"Assign a public IPv4 address"** 가 체크돼 있는지 확인

### 3-3. 공인 IP 확인 (비어 있으면 추가)
인스턴스 상세 → **Public IP address** 확인.
⚠️ 비어 있으면 → 인스턴스 → Attached VNICs → VNIC 클릭 → **IP administration** 탭 → 사설IP 행의 **⋮** → Edit → **Public IP type 을 Ephemeral 로 변경** → 저장.

> 공인 IP는 기본 **Ephemeral(임시)** 이라 인스턴스 정지/재시작 시 바뀔 수 있다. 고정하려면 Reserved IP로 전환.

## 4. 서버 접속 (Windows PowerShell)

PowerShell을 열고 (키 경로·IP는 본인 것으로):
```powershell
ssh -i "C:\Users\user\Downloads\본인키.key" ubuntu@서버공인IP
```
- 처음 접속 시 `(yes/no)` → `yes`
- 프롬프트가 `ubuntu@...:~$` 로 바뀌면 서버 접속 성공

⚠️ **키 권한 에러**(`UNPROTECTED PRIVATE KEY FILE`)가 나면 먼저 실행 후 다시 접속:
```powershell
icacls "C:\Users\user\Downloads\본인키.key" /inheritance:r
icacls "C:\Users\user\Downloads\본인키.key" /grant:r "$($env:USERNAME):R"
```

> 이후 5~8단계는 모두 **서버에 접속한 상태**(`ubuntu@...:~$`)에서 리눅스 명령으로 실행한다.

## 6. 서비스키 넣기 (g2b·iris API용, 선택)

data.go.kr 인증키를 서버에 파일로 둔다(깃에는 안 올라감).
```bash
nano service_key.txt
```
- 편집창에 **키 값**을 붙여넣기(우클릭 또는 Shift+Insert) → `Ctrl+O` → `Enter` → `Ctrl+X`
- 확인: `cat service_key.txt` → 키 문자열이 보여야 함

⚠️ **자주 하는 실수** — 파일 **경로**(`C:\...\service_key.txt`)가 아니라 파일 **안의 키 값**을 넣어야 한다. (PC에서 메모장으로 파일을 열어 내용을 복사)
⚠️ data.go.kr 키는 인코딩/디코딩 두 종류가 있는데, 코드가 자동 처리하므로 **둘 중 아무거나** 넣어도 된다.

## 7. 자동 실행 등록 (systemd)

폴더 경로가 `S_Crawling` 인 기준. **통째로 복사**해 실행:
```bash
sudo tee /etc/systemd/system/siren.service >/dev/null <<'EOF'
[Unit]
Description=Siren Crawling Dashboard
After=network.target

[Service]
Type=simple
User=ubuntu
WorkingDirectory=/home/ubuntu/S_Crawling
ExecStart=/home/ubuntu/S_Crawling/.venv/bin/gunicorn --workers 1 --threads 4 --timeout 120 --bind 0.0.0.0:5000 wsgi:app
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable --now siren
sudo systemctl status siren        # active (running) 이면 성공 (q로 나옴)
```

⚠️ `systemctl` 의 끝 글자는 **소문자 L**(숫자 1 아님), `daemon-reload`는 **하이픈**.
⚠️ `--workers 1` 필수 — 워커가 여러 개면 스케줄러가 중복돼 하루에 여러 번 수집된다.

## 8. 방화벽 열기 — 2곳 모두 (port 5000)

⚠️ **두 곳 다 열어야** 접속된다. 하나만 열면 `ERR_CONNECTION_TIMED_OUT`.

### 8-1. OCI 보안목록 (클라우드 콘솔)
인스턴스 → Virtual cloud network(VCN) → **Security Lists** → Default Security List → **Security rules** 탭 → **Add Ingress Rules**
- Source Type: **CIDR**
- Source CIDR: **`0.0.0.0/0`**
- IP Protocol: **TCP**
- **Destination Port Range: `5000`** ← ⚠️ Source Port 가 아니라 **Destination** 칸에!

### 8-2. 서버 내부 방화벽 (iptables)
⚠️ 5000 허용 규칙이 반드시 `REJECT ... icmp-host-prohibited` 줄보다 **위**에 있어야 한다.
```bash
# REJECT 줄 번호 확인 (보통 5 또는 6)
sudo iptables -L INPUT -n --line-numbers
# REJECT 가 5번이면 그 앞(5)에 삽입
sudo iptables -I INPUT 5 -p tcp --dport 5000 -j ACCEPT
sudo netfilter-persistent save
# 다시 확인 — 5000 ACCEPT 가 REJECT 보다 위인지
sudo iptables -L INPUT -n --line-numbers
```
> `netfilter-persistent: command not found` 면 `sudo apt install -y iptables-persistent` 후 save.

## 10. 운영 방법

### 코드 업데이트 배포 (앞으로의 표준 절차)
PC에서 코드 수정·푸시 후, 서버에서:
```bash
cd ~/S_Crawling
git pull
sudo systemctl restart siren
```

### 자주 쓰는 명령 (서버에서)
```bash
sudo systemctl status siren     # 상태 (q로 나옴)
sudo systemctl restart siren    # 재시작 (키·코드 바꾼 뒤)
sudo systemctl stop siren       # 중지
journalctl -u siren -f          # 실시간 로그 (Ctrl+C로 나옴)
```

### 키워드 관리
대시보드 화면에서 직접 추가/삭제(즉시 반영). 좁은 키워드는 매칭이 드물 수 있어, 더 자주 보려면 관련어(소음·진동·음향·스마트홈 등)를 넓혀 등록한다.

## 11. 자주 겪는 문제 (Troubleshooting)

| 증상 | 원인·해결 |
|------|-----------|
| 사이트 접속 안 됨(timeout) | 방화벽 2곳 확인. 특히 OCI에서 **Destination** Port 5000 인지, iptables 5000이 **REJECT보다 위**인지 |
| 접속이 "거부됨"(refused) | 서비스가 안 떠 있음 → `sudo systemctl status siren` |
| 공고가 하나도 안 뜸 | 정상일 수 있음 — 키워드가 좁아 오늘 매칭이 0건. `기술개발` 등 흔한 단어로 테스트해 카드가 뜨면 시스템 정상 |
| g2b만 0건 / ReadTimeout | `service_key.txt` 키 확인. data.go.kr API가 느릴 때 가끔 발생(타임아웃 30초로 대응됨) |
| 시간이 9시간 밀려 보임 | (구버전) 서버 UTC 표시 문제 — 최신 코드는 한국시간(KST)으로 표시. `git pull` 후 재시작 |
| 자동 수집이 두 번 됨 | gunicorn 워커가 1개인지 확인 |
| 서버 재시작 후 IP 바뀜 | Ephemeral IP 특성. Reserved IP로 고정 가능 |

## 12. 폴더·파일 구성 요약

```
S_Crawling/
├── app.py              # Flask 웹서버 (페이지 + API)
├── crawler.py          # 5개 사이트 + 핫딜 크롤러
├── scheduler.py        # 매일 08:00 자동 수집(APScheduler)
├── database.py         # SQLite 저장 (시간은 KST)
├── config.py           # 키워드·사이트·설정값
├── wsgi.py             # gunicorn 진입점(스케줄러 기동)
├── requirements.txt    # 기본 패키지
├── requirements-deploy.txt  # 서버용(gunicorn)
├── siren.service       # systemd 유닛(참고용)
├── templates/          # index.html(공고), hotdeal.html(핫딜)
├── static/style.css    # 스타일
├── service_key.txt     # data.go.kr 키 (깃 제외, 서버에서 직접 생성)
└── siren.db            # SQLite DB (깃 제외, 자동 생성)
```

Python · Flask · APScheduler · SQLite · requests/BeautifulSoup · gunicorn(서버)
